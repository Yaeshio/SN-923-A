# 3Dプリント製造管理アプリ UI/UX 課題と改善案

## 1. 課題の概要
現在のアプリケーション（`ImportForm`, `OrderForm`, `StatusBoard`）は基本的な機能（フォームの送信とデータ表示）を網羅しているものの、実運用環境（製造現場での入力作業や、部品点数・設備数が多い状況）での利用を想定した設計になっておらず、UXに多くの課題を抱えています。最小限の構成（MVPレベル）に留まっているため、実運用に耐えうるレベルへ引き上げる必要があります。

## 2. 各画面のUI・UXの課題と改善案

### 2.1. 設計データ登録画面 (`ImportForm`)
**現状の課題:**
* **手入力によるエラーの誘発:** プロジェクトIDとユニットIDの入力欄が手入力用のテキストボックス（`<input type="text">`）になっており、ユーザーは正確なIDを手入力またはペーストする必要があります。これにより、タイポによるエラーが頻発します。
* **フィードバックの欠如:** インポート処理中の状態はボタンが「処理中...」となるだけであり、アップロードの進捗や、結果のプレビュー表示がありません。

**改善案:**
* **検索付きドロップダウンの導入:** プロジェクトやユニットをマスタから検索して選択できるUI（Combobox）に変更します。
* **ドラッグ＆ドロップ対応:** `<input type="file">`のボタンだけでなく、領域全体をターゲットとしたドラッグ&ドロップによるファイル追加UIを実装します。
* **3Dプレビュー機能:** アップロード予定のSTLモデルをThree.js等を用いて登録前に画面上でプレビューできるようにし、視覚的な確認プロセスを追加します。

### 2.2. 製造指示画面 (`OrderForm`)
**現状の課題:**
* **大規模データへの非対応:** 部品（Part）と製造機（Machine）の選択がネイティブの`<select>`要素です。部品数が数百種類に及んだ場合、スクロールして目的の項目を探すのが極めて困難になります。
* **付加情報の不足:** 製造機リストに「名前」と「種別」しか表示されず、現在の「稼働状態（空き、使用中、メンテナンス中など）」が不明なため、誤って稼働中の機械にアサインしてしまうリスクがあります。

**改善案:**
* **サジェスト・フィルタ付きコンポーネントへの刷新:** 検索ボックス付きの選択コンポーネント（React Selectなど）を導入し、型番などの部分一致で部品を探せるようにします。
* **リッチなリスト情報とステータス表示:** 部品のサムネイル画像や、製造機の現在のステータス（カラーバッジ付きで「稼働中」「空き」を表示）を選択UI内に埋め込み、直感的な判断を支援します。

### 2.3. 工程ボード画面 (`StatusBoard`)
**現状の課題:**
* **情報量増加時の視認性低下:** 全ての個体（PartItem）が均一なカードとして並ぶため、アイテム数が増大すると「今対応すべき異常項目」などが埋もれてしまいます。フィルタリングやソート機能もありません。
* **誤操作の誘発:** 各カードのセレクトボックスを切り替えるだけで即座にDBへステータス更新が走ります。「DISCARD（破棄）」や「SHIPPED（出荷）」などの重大な変更に際して、確認ダイアログが存在しません。
* **一括操作（バルク処理）の欠如:** 複数個体を同時に次の工程へ進める場合でも、一つずつセレクトボックスを変更する必要があり、現場での作業負荷が著しく高くなります。

**改善案:**
* **高度な絞り込み・検索機能:** 画面上部に、ステータス、部品名、製造機といった条件での多段フィルタリングと、ソート機能を追加します。
* **カンバンボード型UIの導入:** 現在のカードグリッド方式に加えて、ステータスごと（Todo, In Progress, Completed等）の列を持ち、ドラッグ＆ドロップで直感的に遷移させられるカンバン形式（Trelloライク）の表示モードを提供します。
* **一括操作と確認ダイアログ:** テーブル表示モードや複数選択チェックボックスを追加し、一括でステータス変更できるUIを実装します。また、重大なステータス変更時には理由入力を求めるモーダルを挟むよう改善します。

### 2.4. プロジェクトおよびユニット管理UI（新規追加の必要性）
**現状の課題:**
* **マスタ管理機能の欠如:** データベース上には「プロジェクト（Project）」および「ユニット（Unit）」のテーブルが存在し、設計データのインポート時などにそのIDが要求されます。しかし、現在のアプリケーション上にはこれらのマスタ情報を登録・編集・閲覧・削除（CRUD）するための画面が存在しません。
* **ユーザー単独での運用困難:** マスタ管理UIがないため、システム導入時や新しいプロジェクト・ユニットの発生時に、開発者がデータベースを直接操作（SQL発行等）してレコードを作る必要があり、現場ユーザーのみで運用を回すことが不可能です。

**改善案:**
* **マスタ設定画面群の新設:** サイドナビゲーション等に「設定」あるいは「マスタ管理」メニューを追加し、その配下に「プロジェクト管理」「ユニット管理」の各画面を新規作成します。
* **基本的なCRUDフォームの提供:** テーブルによる一覧表示機能、および新規作成・編集・削除ができるフォーム（モーダルダイアログ、もしくは専用の入力ページ）を提供します。
* **階層構造を意識したUI:** ユニットは特定のプロジェクトに紐付くという業務構造に合わせて、プロジェクトの詳細画面から直接そのプロジェクトにおけるユニットを追加・編集できるような、ナビゲーションに配慮した設計とします。

## 3. 全体的なUX方針の提案
* **レスポンシブと現場端末への最適化:** タブレットや工場内共有タッチパネル端末からの操作を想定し、タップ領域（ボタンや選択肢）を大きく確保したタッチファーストな設計を取り入れます。
* **リアルタイム同期:** 複数人が同時に操作する環境を想定し、他ユーザーの実行したステータス変更が画面リロードなしに即座に反映される仕組み（Supabase Realtime等）を導入します。
